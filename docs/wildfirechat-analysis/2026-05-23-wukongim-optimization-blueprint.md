# WuKongIM 优化蓝图

报告日期：2026-05-23
基于材料：WuKongIM 与 WildfireChat 深度对比报告、既有全链路优化计划、剩余优化收口报告、生产主机压测调优说明。
读者：负责 WuKongIM 后续产品化、性能扩展和生产运维的工程负责人。
读后动作：把本蓝图拆成阶段性工程包，建立可验证的性能、稳定性、安全和产品能力提升闭环。

## 1. 总体判断

你的 WuKongIM 路线应该继续推进，不建议整体迁移到 WildfireChat。原因是当前项目已经围绕 WuKongIM 内核、TangSengDaoDaoServer、Flutter 多端客户端、Nginx、MinIO、LiveKit、Coturn、MySQL、Redis 建立了自己的生产工程闭环；整体换栈会带来 SDK、消息模型、登录 token、音视频、文件、历史消息和客户端本地库的大规模迁移成本。

但当前 WuKongIM 仍然处在“可运行的单节点产品栈”向“可验证、可扩展、可审计的 IM 平台”演进的阶段。与 WildfireChat 相比，主要差距不是能不能聊天，而是生产化外围能力和工程治理：多节点高可用、标准压测基线、全链路观测、厂商推送、归档审计、企业组织、开放平台、会议录制、发布门禁和长期运维体系。

优化目标应从“继续加功能”转为“把能力变成可证明的系统指标”。每一项优化都要有基线、有灰度、有回滚、有验收，而不是只看代码是否写完。

## 2. 当前基线

### 2.1 已经具备的优势

当前 WuKongIM 项目已经具备以下基础：

- Flutter 多端客户端已经覆盖 Android、Web、Windows 等主要形态，并完成了大量本地 SDK、PWA、通知、文件、图片、拖拽、扫码、媒体缓存和聊天 viewport 优化。
- TangSengDaoDaoServer 已经承担用户、群、消息、文件、机器人、搜索、监控、实时 session 和 callgateway 等业务能力。
- 生产部署已经通过 Nginx 收口 HTTP/WSS 入口，WuKongIM 管理/API 端口已收敛到本地绑定，安全姿态比裸露 IM 端口更好。
- 消息本地搜索、SQLite WAL、FTS5、Map 化装配、viewport 索引化、媒体 L1 字节预算等性能优化已经有代码和测试证据。
- 客户端 protobuf control codec 已经支持未来 `event_id`、`aggregate_id`、`schema_version` 等协议字段，为实时事件 v2 铺好了前向兼容基础。
- 已经有 production doctor、smoke、perf probe、备份恢复、主机 sysctl/nofile 调优 runbook 等运维脚本基础。

### 2.2 仍然存在的关键短板

当前最需要补齐的短板如下：

- 生产部署仍然是单节点 WuKongIM 栈，没有正式启用 WuKongIM 多节点集群和故障切换。
- 缺少独立压测环境和固定性能基线，无法证明当前部署能承受多少连接、多少 send QPS、多少群 fanout、多少历史分页和搜索压力。
- Prometheus/Grafana 观测闭环不完整，Nginx、WuKongIM、TSDD、MySQL、Redis、MinIO、LiveKit、主机资源之间缺少统一看板和告警。
- MySQL、Redis、MinIO、LiveKit、Coturn、WuKongIM、TSDD 仍然共用一台约 7.5 GiB 内存服务器，压测时会出现 CPU、内存、磁盘 IO、网络带宽互相干扰。
- Android 厂商推送体系不完整，中国大陆 Android 离线到达率会明显弱于 WildfireChat 的 push_server 生态。
- 归档、审计、长期历史搜索、企业组织、开放平台、频道平台、会议录制等外围产品能力还没有形成成熟平台。
- 实时事件总线、服务端 proto v2、Repository 分层、Web IndexedDB、Windows DB worker、大文件分片续传、E2EE、桌面系统能力等仍是跨端跨服务的大工程。

## 3. 优化原则

1. 先建基线，再做优化。没有基线的优化无法证明收益，也无法定位回退点。
2. 先稳定单节点，再推进多节点。单节点的发布、备份、观测、限流、压测没有闭环时，直接上集群会扩大故障面。
3. 先做协议兼容，再做协议替换。实时事件 v2、ack、gap repair、event id 去重必须支持灰度和回滚。
4. 核心链路轻量化，重任务异步化。长连接只承载实时通知和轻量命令，历史拉取、搜索、文件、转码、审计归档走 HTTP 或异步任务。
5. 外围能力平台化。厂商推送、归档审计、组织、开放平台不能只做业务点，应按可复用服务设计。
6. 所有生产变更必须可观测、可回滚、可演练。尤其是端口、secret、数据库迁移、协议升级、集群切换和主机调优。

## 4. 30/60/90/180 天路线图

### 4.1 0-30 天：稳定性、基线和风险收口

目标：把当前单节点生产栈打磨成可观测、可压测、可回滚的稳定版本。

重点任务：

- 建立独立压测环境，不在生产环境开启 bench API。
- 固定压测场景：长连接数、单聊 send QPS、小群 fanout、大群 fanout、断线重连、历史分页、搜索、文件上传、WebSocket 重连。
- 接入基础 Prometheus/Grafana：主机、Docker、Nginx、WuKongIM、TSDD、MySQL、Redis、MinIO、LiveKit。
- 对 `/ws` 做握手限流、异常协议采样、连接失败分类、黑名单或临时封禁策略。
- 做 secret 管理和脱敏检查：`.env`、渲染 YAML、日志、备份、CI 产物、文档都不得泄漏明文密钥。
- 完成备份恢复演练：MySQL、Redis、MinIO、WuKongIM 数据、TSDD 数据。
- 把 production doctor、smoke、perf probe 固化为发布门禁。
- 明确当前服务器容量红线：CPU、内存、磁盘 IO、网络、文件描述符、conntrack、Redis AOF、MySQL buffer pool。

验收指标建议：

- 每次发布都有 smoke 和 perf probe 结果记录。
- `/ws` 异常握手不会持续打入 WuKongIM 协议层造成日志噪声。
- 能在非生产环境复现并记录至少三类性能基线：连接、发送、历史查询。
- 备份恢复演练可以在预演环境完整恢复核心业务数据。
- 生产配置、日志和文档通过敏感信息扫描。

### 4.2 31-60 天：协议、存储和客户端性能工程化

目标：把前期已经做了一半的实时协议、存储和跨端优化补成完整工程包。

重点任务：

- 服务端 proto v2：在 Go 源 proto 和生成代码中加入 `event_id`、`aggregate_id`、`schema_version`，保持旧字段兼容。
- 建立完整 session event bus：纳入 `message.new`、`message.extra.changed`、`conversation.changed`、`call.invited` 等核心事件。
- 完成 ack/gap repair 观测：按设备、平台、版本、事件类型统计 gap、重复、decode error、ack lag。
- 推进 Repository 分层：将消息、文件、搜索、会话、平台能力从 UI 和 IMService 中继续拆分。
- 完成 Web IndexedDB 近端缓存：缓存最近会话和最近消息页，降低刷新后的远程依赖。
- 完成 Windows DB worker/isolate：大批量搜索、导入、备份恢复不阻塞 UI。
- 大文件普通上传链路继续流式化，并规划分片上传接口和客户端恢复状态。
- 把当前 FTS5、WAL、Map 装配、viewport 索引化等优化纳入长期回归测试。

验收指标建议：

- protobuf 控制协议灰度到 100% 后 decode error rate 低于 0.1%。
- 断线 30 秒内恢复后无重复消息、无消息洞，gap repair 有指标可查。
- 10 万条本地消息的频道内搜索 p95 进入可接受区间，建议目标小于 300 ms。
- Windows 搜索和批量导入期间 UI 不出现长时间卡死。
- Web 刷新后最近会话首屏不完全依赖远程历史接口。

### 4.3 61-90 天：高可用和资源拆分

目标：验证 WuKongIM 多节点能力，降低单机资源竞争和单点故障风险。

重点任务：

- 搭建三节点 WuKongIM staging cluster，验证 slot/channel 副本、leader 转移、节点宕机、滚动升级、网络抖动。
- 设计生产集群迁移方案：先灰度 staging，再小流量验证，再迁移主生产。
- TSDD API 做无状态化检查，确认多实例部署需要的 Redis/session/cache 边界。
- 将 MySQL、Redis、MinIO 至少一个组件迁出同机，优先迁出数据库或对象存储，降低磁盘 IO 竞争。
- LiveKit/Coturn 独立规划公网 IP、带宽、UDP/TCP 暴露和指标，不再与 IM/API 争抢资源。
- Nginx upstream 加健康检查、failover、连接数和错误率监控。
- 建立故障演练：IM 节点宕机、TSDD 实例宕机、Redis 重启、MySQL 慢查询、MinIO 不可用、LiveKit 不可用。

验收指标建议：

- staging 集群可以完成单节点停止和恢复演练，客户端能够重连并继续收发。
- TSDD 至少可跑两个实例，API 层无本地状态依赖。
- 数据库或对象存储迁出后，压测期间 IM/API 与存储 IO 竞争明显下降。
- 生产发布可完成滚动升级预演，不再依赖全量停机重启。

### 4.4 91-180 天：产品生态补齐

目标：补齐与 WildfireChat 相比最明显的外围生产能力，让 WuKongIM 不只是通信内核，而是完整 IM 平台。

重点任务：

- 建立独立 push service：支持 APNs、FCM、华为、荣耀、小米、OPPO、vivo 等通道，统一设备 token、厂商回执、失败重试和到达率统计。
- 建立 archive service：通过 messageEffects 或消息同步链路将消息、extra、会话变更写入 ClickHouse 或等价分析存储，用于归档、审计和长期搜索。
- 建立组织平台：部门、员工、组织树、部门群、导入导出、权限边界、离职转移。
- 建立开放平台：第三方应用注册、app secret、机器人、webhook、频道、事件订阅、回调签名、限流和审计。
- 建立管理控制台：统一管理用户、群、设备、消息审计、推送、归档、敏感词、机器人、系统健康、告警。
- 完善会议能力：LiveKit 会议录制、回放、纪要、权限、存储生命周期和成本控制。
- 评估 E2EE 私聊试点：设备信任、密钥备份、新设备迁移、不可搜索提示、合规边界。

验收指标建议：

- Android 离线推送覆盖主流厂商，并有到达率、点击率、失败原因统计。
- 归档链路可按用户、群、时间范围、关键词检索，并有权限校验。
- 管理端能追踪一次消息从发送、投递、离线推送到归档的完整链路。
- 第三方应用和机器人不再共享内部管理密钥，所有回调都有签名和审计。
- 会议录制和回放不影响 IM 主链路性能。

## 5. 分领域优化蓝图

### 5.1 安全和生产稳定性

当前最重要的不是增加复杂功能，而是把生产暴露面、密钥、备份和发布流程收紧。

建议动作：

- 公网入口只保留 80/443 和 RTC/TURN 必需端口。WuKongIM 原始 TCP、WebSocket、manager/API、MySQL、Redis、MinIO console 等不应公网暴露。
- `/ws` 入口增加连接速率限制、握手失败分类、User-Agent/IP 采样、异常帧计数、连接时长分布。
- 所有 secret 只进入受控配置源，日志、备份、CI、文档统一脱敏。
- `sms.super_code` 这类测试便捷能力必须有环境开关，生产默认关闭或只在白名单环境可用。
- 文件上传限制大小、类型、内容长度、超时和并发，MinIO 桶策略避免匿名写和不必要删除权限。
- 所有生产脚本保留 dry-run、回滚和变更记录。

预期提升：

- 降低被扫描、撞库、异常协议消耗和配置泄露导致的生产事故概率。
- 让故障排查从“看日志猜”变成“看入口、应用、存储、主机指标定位”。
- 让每次发布和配置变更都有可回退路径，减少人为操作风险。

### 5.2 性能基线和观测

WuKongIM 内核性能潜力高，但你的生产栈真正能承受多少，需要用独立基线证明。

建议动作：

- 建立固定压测项目：连接保持、登录、单聊发送、群 fanout、历史分页、搜索、文件上传、离线重连。
- 每次性能测试记录环境：机器规格、内核参数、Docker 限制、WuKongIM/TSDD 镜像版本、数据库大小、Redis 策略、网络带宽。
- 观测指标至少覆盖：
  - Nginx：QPS、状态码、WebSocket 101、4xx/5xx、upstream latency、连接数。
  - WuKongIM：连接数、send QPS、sendack p95、recvack、message seq lag、节点状态、Raft/slot 状态。
  - TSDD：接口 p95/p99、慢 SQL、错误率、goroutine、GC、消息同步耗时。
  - MySQL：QPS、慢查询、buffer pool、连接数、锁等待、磁盘 IO。
  - Redis：内存、AOF、blocked clients、ops、keyspace、延迟。
  - MinIO：上传下载吞吐、错误率、磁盘空间、对象数量。
  - LiveKit/Coturn：房间数、参与者、码率、丢包、TURN 流量。
  - 主机：CPU、load、内存、swap、磁盘、网卡、nofile、conntrack、TCP 状态。

预期提升：

- 能清楚回答“当前版本能扛多少用户、多少连接、多少消息”。
- 性能退化能在发布前发现，而不是在线上用户投诉后发现。
- 优化收益可以量化，例如发送 p95、搜索 p95、重连成功率、文件上传内存峰值、慢 SQL 数量。

### 5.3 实时协议和事件总线

当前客户端已经具备 protobuf v2 字段兼容基础，下一步要把服务端和全链路事件模型补齐。

建议动作：

- 将 `event_id` 作为事件幂等键，避免弱网、重试、gap repair 造成重复处理。
- 将 `aggregate_id` 用于会话、消息、设备、通话等聚合对象追踪。
- 将 `schema_version` 作为灰度、回滚和跨版本兼容判断依据。
- 长连接只推事件通知，不承载重数据；客户端收到事件后按需用 HTTP 拉取消息、extra、会话、通话详情。
- 统一事件类型命名，至少覆盖消息、消息扩展、会话变更、通话邀请、设备失效、配置变更。
- 对每类事件建立 ack lag、gap count、duplicate count、decode error、pull fallback 指标。

预期提升：

- 弱网和多端场景下消息补洞更稳定。
- 协议升级可以灰度，不再一次性切换全量用户。
- 事件去重、审计、归档和开放平台订阅都有统一语义。

### 5.4 存储、历史和搜索

IM 产品长期体验取决于历史消息、搜索、归档和本地库稳定性。

建议动作：

- 客户端继续坚持 SQLite WAL、FTS5、busy_timeout、Map 装配、索引化 viewport。
- Web 增加 IndexedDB 近端消息页缓存，降低刷新后首屏和最近历史的远程依赖。
- Windows 将大查询、批量导入、备份恢复、归档解析移到 worker/isolate。
- 服务端对消息、message_extra、reaction、conversation_extra 等增量接口建立 EXPLAIN 和慢查询门禁。
- 冷热历史分层：近期消息走主链路，长期归档进入 ClickHouse/ES/OpenSearch 等分析检索存储。
- 归档链路不阻塞主消息发送，采用异步消费、失败重试和补偿扫描。

预期提升：

- 十万级本地历史和复杂 reaction/extra 场景下客户端更流畅。
- Web/Windows 体验接近桌面 IM，而不是每次都重新拉远端。
- 合规审计、长期搜索、运营排障不再直接冲击主业务数据库。

### 5.5 高可用和横向扩展

WuKongIM 官方内核有多节点和分布式设计，但当前生产没有吃到这部分红利。

建议动作：

- 先在 staging 搭三节点 WuKongIM cluster，演练节点宕机、网络抖动、leader 转移、slot 迁移、滚动升级。
- 建立集群压测基线，不只测单节点吞吐，还要测故障期间的可用性和恢复时间。
- TSDD API 多实例化，所有 session/cache/临时状态落 Redis 或外部存储。
- 数据库、对象存储、RTC 逐步从 IM 主机迁出。
- 生产迁移前制定清晰回滚方案：DNS、Nginx upstream、数据目录、配置、镜像版本、客户端兼容。

预期提升：

- 从“单节点故障导致整体中断”提升到“节点故障可切换、可恢复、可演练”。
- 压测时 IM、数据库、对象存储、RTC 互相干扰减少，容量边界更清晰。
- 后续增长可以通过加节点、拆资源、扩实例处理，而不是只升级单机规格。

### 5.6 推送服务

这是与 WildfireChat 生态差距最大的实际生产能力之一。

建议动作：

- 设计独立 push service，不把厂商推送逻辑散落在 TSDD 或客户端临时逻辑里。
- 统一设备 token 表：uid、device_id、platform、vendor、token、app_version、locale、last_seen、status。
- 支持 APNs、FCM、华为、荣耀、小米、OPPO、vivo 等通道，后续可接入统一推送供应商。
- 建立推送任务表和回执表：创建、发送、成功、失败、失效 token、限流、重试。
- 支持消息类、通话类、系统类推送模板，避免 payload 里泄漏敏感内容。
- 将 LiveKit 通话邀请、弱网离线补偿、普通离线消息纳入统一推送策略。

预期提升：

- 中国大陆 Android 离线消息到达率显著提升。
- 通话邀请成功率更高，后台来电体验更接近原生 IM。
- 可以通过指标区分“IM 未投递”“厂商未到达”“用户未点击”“token 失效”等问题。

### 5.7 归档、审计和风控

WildfireChat 的 archive-server、anti-fraud 给出的启发是：消息平台需要独立的审计链路，不能只依赖在线消息表。

建议动作：

- 通过 messageEffects 或消息同步链路生成归档事件。
- 将消息、撤回、编辑、reaction、已读、群成员变化、敏感词命中写入归档存储。
- 建立审计查询 API：按用户、群、时间、关键词、消息类型、操作类型检索。
- 查询必须做权限校验，普通用户只能查自己有权访问的范围，管理员查询保留审计日志。
- 敏感词、反诈、违规内容处理走异步事件，不阻塞主发送链路。
- 对 E2EE 消息明确产品边界：服务端不可搜索、不可内容审计，只能审计 metadata。

预期提升：

- 管理和合规能力明显增强，适合企业客户或受监管场景。
- 历史搜索和排障不会拖慢主 MySQL。
- 风控命中、管理员处理、用户通知形成闭环。

### 5.8 组织、开放平台和管理端

如果目标是企业 IM 或可扩展 IM 平台，这部分必须补齐。

建议动作：

- 组织平台：组织树、部门、员工、岗位、部门群、导入导出、禁用离职、权限继承。
- 开放平台：第三方应用、机器人、webhook、事件订阅、app secret、签名、限流、IP 白名单。
- 频道平台：公众号/频道身份、菜单、自动回复、订阅者、群发、素材。
- 管理控制台：用户、群、设备、消息审计、推送、归档、机器人、敏感词、系统健康、发布版本。
- 权限模型先做清楚：超级管理员、组织管理员、审计员、应用管理员、普通成员。

预期提升：

- 从“单一 IM 产品”升级为“可被企业和第三方接入的平台”。
- 机器人和 webhook 能形成生态，不需要每次都直接改 TSDD 业务代码。
- 管理和审计能力可销售、可交付、可运维。

### 5.9 文件、媒体和会议

文件和音视频是最容易拖垮 IM 主链路的能力，必须独立治理。

建议动作：

- 普通上传保持流式化，避免完整文件进入 API 内存。
- 大文件增加分片上传：init、upload chunk、complete、abort、hash 校验、断点恢复。
- MinIO 对象生命周期按业务类型设置：头像、图片、普通文件、会议录制、临时转码文件分开策略。
- LiveKit 独立监控房间、参与者、码率、丢包、TURN 流量和录制任务。
- 会议录制、转码、纪要生成全部异步化，不进入 IM/TSDD 主响应链路。

预期提升：

- 大文件并发上传时 API 内存峰值更可控。
- 弱网断点续传减少用户重复上传成本。
- 音视频不会轻易抢占 IM/API 的 CPU、带宽和磁盘资源。

### 5.10 客户端体验和工程治理

Flutter 多端统一是你的优势，但复杂度也会持续上升。

建议动作：

- 继续拆分 Platform Adapter，减少 `dart:io`、Web、Windows、Android 差异逻辑散落。
- 将 IMService 收敛为应用编排层，消息、文件、搜索、会话、通话分别进入 Repository。
- 聊天列表坚持稳定 key、O(1) upsert、局部刷新、骨架屏和弱网不清空已有内容。
- 桌面端补齐系统托盘、全局快捷键、多窗口、未读角标、右键菜单、拖拽发送。
- Web 端完善 PWA、浏览器通知、IndexedDB、离线页、Wasm/CanvasKit 缓存策略。
- Android 端重点补厂商推送、后台来电、权限失败反馈、低内存图片解码策略。

预期提升：

- 多端体验一致性提升，同时降低平台分支带来的回归风险。
- 长历史、弱网、reaction 高频刷新、大图、大文件场景更稳定。
- 桌面和 Web 从“可用”向“日常主力 IM 客户端”靠近。

## 6. 优化后的整体提升预期

### 6.1 可用性提升

当前最大风险是单节点生产栈。一旦 WuKongIM、TSDD、MySQL、Redis、MinIO、LiveKit 所在主机发生故障，整体服务都会受影响。完成高可用和资源拆分后，系统会从“单机可运行”提升到“节点故障可切换、组件故障可降级、恢复过程可演练”。

预期结果：

- IM 节点故障不再等同于全站不可用。
- TSDD 可以多实例部署，API 层具备横向扩展基础。
- 数据库、对象存储、RTC 的故障影响范围更清晰。
- 发布和回滚更接近标准生产流程。

### 6.2 性能容量提升

WuKongIM 内核有高并发潜力，但当前真实瓶颈可能出现在 TSDD、MySQL、Redis、MinIO、LiveKit、Nginx、主机参数或客户端本地库。通过压测、观测和资源拆分，性能提升不是单点“调快”，而是容量边界更清楚、退化更可控。

预期结果：

- 可以明确给出当前版本的连接数、send QPS、群 fanout、历史分页和搜索基线。
- 慢 SQL、Redis AOF、MinIO IO、LiveKit 带宽对 IM 主链路的干扰会下降。
- 本地搜索、聊天列表刷新、历史分页的 p95/p99 更稳定。
- 未来扩容可以根据瓶颈加节点或拆组件，而不是盲目升级服务器。

### 6.3 安全提升

端口收口、secret 管理、日志脱敏、文件权限、Webhook 签名、推送密钥隔离和审计权限会显著降低暴露面。

预期结果：

- 扫描流量、异常协议、弱口令、配置泄漏造成事故的概率降低。
- 管理密钥、厂商推送密钥、机器人密钥不再混杂在普通业务代码和客户端侧。
- 文件上传、归档查询、开放平台回调都有明确鉴权和审计。

### 6.4 用户体验提升

用户能感受到的提升主要来自四个方向：消息更稳、离线推送更准、历史搜索更快、弱网恢复更自然。

预期结果：

- Android 离线消息和来电到达率提升。
- 弱网重连后重复消息、漏消息、状态错乱减少。
- Web/Windows 刷新、搜索、历史加载体验更接近原生客户端。
- 大文件上传可以恢复，图片和媒体加载更不容易卡顿。

### 6.5 运维效率提升

Prometheus/Grafana、发布门禁、doctor、smoke、perf baseline、备份演练和故障演练会让运维从人工经验转向流程化。

预期结果：

- 一次线上问题可以从用户、入口、IM、TSDD、数据库、Redis、对象存储、RTC 逐层定位。
- 发布前能看到性能是否退化。
- 证书、备份、磁盘、连接数、慢查询、推送失败率可以提前告警。
- 新工程师接手系统时有指标和 runbook，而不是依赖口头经验。

### 6.6 商业化和生态提升

推送、归档、组织、开放平台、会议录制、管理端补齐后，WuKongIM 会从“自用聊天系统”向“可交付 IM 平台”升级。

预期结果：

- 面向企业客户时，可以提供组织通讯录、部门群、审计、归档、管理后台。
- 面向开发者时，可以提供机器人、webhook、第三方应用、频道和事件订阅。
- 面向合规场景时，可以提供可检索、可授权、可追踪的消息审计。
- 与 WildfireChat 的生态差距会明显缩小，同时保留 WuKongIM 内核性能和 Flutter 多端统一优势。

## 7. 可量化收益目标

以下目标不是当前生产能力承诺，而是完成对应优化后应当用压测、监控和演练证明的验收方向。正式数值需要在独立压测环境中按机器规格和业务模型校准。

| 维度 | 当前状态 | 优化后目标 | 验证方式 |
| --- | --- | --- | --- |
| 容量认知 | 没有正式独立压测基线，只能根据运行状态和官方资料推断 | 每个版本都有连接数、send QPS、群 fanout、历史分页、搜索、上传基线 | wkbench、业务压测脚本、Grafana 版本标记 |
| 可用性 | 单节点生产栈，主机或核心容器故障会影响整体服务 | staging 先验证三节点 IM 集群，生产逐步支持节点故障切换 | 宕机演练、滚动升级演练、客户端重连指标 |
| 发布质量 | 已有 doctor/smoke/perf 脚本基础，但门禁还需固化 | 发布前必须通过 smoke、核心 API、WSS 握手、基础性能探针 | CI/CD 或发布脚本强制检查 |
| 弱网一致性 | protobuf 客户端兼容已具备，服务端事件模型未闭环 | 事件可去重、可 ack、可 gap repair、可按版本灰度回滚 | decode error、ack lag、gap count、duplicate count |
| 历史搜索 | SDK FTS5/WAL 等基础已完成，Web/Windows 分层仍需推进 | 移动端十万级历史可搜索，Web 有近端缓存，Windows 大查询不阻塞 UI | 客户端专项 benchmark、UI 帧耗时、搜索 p95 |
| 安全暴露面 | 入口已明显收口，但 secret、文件、回调、推送仍需制度化治理 | 配置不泄漏、入口有限流、文件权限最小化、回调签名、管理查询可审计 | 敏感信息扫描、端口扫描、权限审计、日志审计 |
| 资源隔离 | 多个重组件共用一台 7.5 GiB 级别主机 | 数据库、对象存储、RTC 或 IM 至少逐步拆分，瓶颈互相隔离 | 压测期间 CPU/IO/网络/内存指标对比 |
| 离线到达 | Android 厂商推送体系不完整 | 主流厂商通道接入，能统计到达、点击、失败和 token 失效 | push service 回执、厂商错误码、客户端点击回传 |
| 审计归档 | 长期归档和合规检索未平台化 | 消息、变更、风控命中进入异步归档，查询有权限和审计 | 归档延迟、补偿任务、查询权限测试 |
| 商业化能力 | 当前更像面向自身业务的 IM 产品 | 形成组织、开放平台、管理端、会议录制等可交付能力 | 管理端功能验收、第三方应用接入测试 |

## 9. 推荐实施顺序

第一阶段不要先做大规模重构，也不要直接上生产集群。先把当前单节点打造成“有指标、有备份、有门禁、有容量答案”的稳定形态。

推荐顺序：

1. 建立压测环境和 Grafana 基线。
2. 收口安全暴露面和 secret 管理。
3. 固化发布门禁、备份恢复和 smoke/perf probe。
4. 补齐服务端 proto v2 和 session event bus。
5. 推进 Repository、Web IndexedDB、Windows worker 等客户端结构化优化。
6. 在 staging 验证三节点 WuKongIM cluster。
7. 迁出数据库、对象存储或 RTC，降低同机资源竞争。
8. 建 push service 和 archive service。
9. 再做组织、开放平台、管理端、会议录制、E2EE 等平台化能力。

## 10. 最大风险和回滚策略

### 10.1 协议升级风险

风险：新老客户端并存时，事件字段、ack、gap repair、payload 解析不兼容，可能导致重复消息、漏消息或会话状态错乱。

回滚策略：

- 保留 JSON/旧协议回退。
- 按用户、设备、版本灰度。
- 所有事件保留 `schema_version`。
- 服务端保留双写或兼容读取窗口。
- 以 decode error、gap count、duplicate count、ack lag 作为自动回滚信号。

### 10.2 集群迁移风险

风险：slot/channel 迁移、节点故障、滚动升级、客户端重连策略没有验证充分，生产切换可能造成连接抖动或消息延迟。

回滚策略：

- 先 staging 压测，再小流量灰度。
- 保留单节点生产配置和数据回退路径。
- Nginx upstream/DNS 切换可快速回滚。
- 切换期间冻结非必要业务发布。

### 10.3 存储拆分风险

风险：MySQL、Redis、MinIO 迁移可能造成网络延迟、权限、备份、连接池、DNS 和数据一致性问题。

回滚策略：

- 先迁只读或低风险组件。
- 迁移前做全量备份和恢复演练。
- 连接串、账号权限、超时、连接池都有回滚配置。
- 迁移后至少观察一个完整业务周期。

### 10.4 推送和归档风险

风险：推送可能重复、泄漏内容或被厂商限流；归档可能拖慢主链路或产生权限越权查询。

回滚策略：

- 推送 payload 最小化，不携带敏感内容。
- 归档异步化，失败不阻塞发送。
- 查询权限独立校验。
- 所有管理员查询写审计日志。

## 11. 最终建议

未来半年最值得投入的不是替换 WuKongIM，而是把它的平台化短板补齐。短期重点是压测、观测、安全、备份和发布门禁；中期重点是协议事件化、客户端存储分层和 staging 集群；长期重点是推送、归档、组织、开放平台、会议录制和管理端。

完成这些优化后，WuKongIM 的定位会从“基于高性能内核的可运行 IM 产品”升级为“具备高可用、可观测、可扩展、可审计和可商业交付能力的 IM 平台”。这才是与 WildfireChat 竞争时真正需要补齐的部分。
