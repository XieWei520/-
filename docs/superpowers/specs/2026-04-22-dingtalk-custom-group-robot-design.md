# DingTalk Custom Group Robot Design

**Status:** Approved in-thread based on the user's confirmation to implement DingTalk as a separate custom-robot channel parallel to the existing Feishu group robot flow.

## Goal

Add a `钉钉机器人` capability to group management so a group owner or admin can generate a WuKongIM webhook and secret, then let third-party systems push DingTalk custom robot payloads into that IM group.

## Scope

This design covers only DingTalk **custom robot** compatibility:

- group-scoped webhook generation
- group-scoped sign secret generation
- enable or disable the robot
- regenerate webhook token
- regenerate sign secret
- delete config
- send a local test message into the IM group
- receive DingTalk-style webhook payloads and convert them to IM messages

## Non-Goals

This design explicitly does not include:

- DingTalk app robot `appKey/appSecret/OpenAPI`
- sending outbound messages from WuKongIM to DingTalk
- a shared multi-provider robot abstraction across all providers
- manager-side admin console changes in phase 1

## Product Shape

The DingTalk experience should mirror the existing Feishu robot flow in Flutter:

- A new `钉钉机器人` entry is added on the group detail page.
- Opening it lands on a dedicated settings page.
- The page shows the generated webhook URL and sign secret.
- The page offers enable, regenerate, test, and delete actions.
- The page explains that callers should use DingTalk custom robot message format.

## Architecture

The implementation stays intentionally parallel to the existing Feishu integration.

### Backend

Add a new DingTalk robot module under the existing `modules/robot` package with:

- HTTP routes for CRUD and test operations under `/v1/groups/:group_no/robot/dingtalk`
- a webhook receiver under `/v1/groups/:group_no/robot/dingtalk/webhook/:token`
- a dedicated table `robot_dingtalk_group`
- helper functions for token generation, signature validation, payload parsing, and IM payload building

### Flutter

Add a dedicated DingTalk config model, API methods, and a new `GroupDingTalkBotPage`. Wire the page from `group_detail_page.dart` the same way `GroupFeishuBotPage` is wired today.

## Backend Contract

Phase 1 uses these routes:

- `GET /v1/groups/:group_no/robot/dingtalk`
- `PUT /v1/groups/:group_no/robot/dingtalk`
- `DELETE /v1/groups/:group_no/robot/dingtalk`
- `POST /v1/groups/:group_no/robot/dingtalk/test`
- `POST /v1/groups/:group_no/robot/dingtalk/webhook/:token`

The config response mirrors the Feishu response shape where practical:

```json
{
  "group_no": "g1",
  "webhook_url": "https://host/v1/groups/g1/robot/dingtalk/webhook/token_xxx",
  "secret": "SECxxx",
  "enabled": 1,
  "secret_set": true,
  "last_push_at": 1710000000,
  "last_error": "",
  "updated_at": "2026-04-22 12:00:00"
}
```

## Security

The DingTalk webhook receiver validates:

- the path token generated by WuKongIM
- that the config exists and is enabled
- the DingTalk-style signature query parameters when a secret is configured

The supported signature mode for phase 1 is the widely used DingTalk custom robot pattern:

- query params: `timestamp` and `sign`
- signing string: `timestamp + "\n" + secret`
- algorithm: HMAC-SHA256 with the same secret
- digest encoding: base64

If the signature is missing or invalid while a secret is configured, the webhook is rejected.

## Message Compatibility

Phase 1 accepts the DingTalk custom robot message types that give the most coverage without forcing a rich-card renderer:

- `text`
- `markdown`
- `link`
- `actionCard`
- `feedCard`

Mapping rules:

- `text`: map directly to IM text content
- `markdown`: flatten title and markdown body into readable IM text
- `link`: convert to a readable summary with title, text, and URL
- `actionCard`: convert to a readable summary with title, body, and button information
- `feedCard`: convert link items into a readable multi-line summary

Unsupported or malformed payloads return a structured error and do not emit group messages.

## Error Handling

The receiver updates `last_error` when:

- the payload is invalid
- signature validation fails
- payload mapping fails
- IM delivery fails

On success it clears `last_error` and updates `last_push_at`.

## Testing Strategy

The implementation is backend-first and test-first:

1. Add Go tests for DingTalk token extraction, signature validation, webhook URL generation, and message text resolution.
2. Implement the backend until those tests pass.
3. Add Flutter tests for the new group-detail entry, API methods, and DingTalk settings page labels and actions.
4. Implement Flutter until those tests pass.
5. Rebuild and relaunch the Windows desktop app for manual QA.

## Risks

- DingTalk docs are dynamically rendered, so phase 1 deliberately targets the stable and common custom-robot webhook contract rather than app-bot OpenAPI behavior.
- Payload compatibility is text-first. This is intentional to ship a reliable bridge without inventing a DingTalk card renderer inside the IM client.
- The workspace snapshot is not a normal git worktree, so the design doc is saved locally but cannot be committed from this environment.
