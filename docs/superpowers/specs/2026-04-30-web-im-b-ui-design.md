# WuKong IM Web B UI 设计规格

日期：2026-04-30

## 结论

采用已确认的 B 方向：温暖社交感。范围采用 Web 核心全套，覆盖登录、主壳层、会话列表、聊天页、通讯录、个人中心、来电/操作弹层。移动端、桌面端和消息协议不在本轮重写范围内。

本轮目标是让 Web 端从“移动端页面直接铺到浏览器”升级为“浏览器 IM 工作台”。视觉更温暖、社交、清晰，但信息密度仍服务聊天效率。

## 设计目标

1. Web 首屏像一个完整 IM 工作台，宽屏有稳定导航、会话列表、聊天内容和上下文区域。
2. 保留既有消息、会话、撤回、emoji、头像、音视频通话等业务能力，只替换壳层、主题和核心组件视觉。
3. 降低旧灰色界面的割裂感，用统一的颜色、间距、圆角、阴影和状态样式贯穿 Web 核心路径。
4. 优先保证聊天可读性、操作可达性和状态反馈，不做营销页式大图或装饰性布局。
5. 保持 Android/iOS/Windows 既有体验，Web 专属布局通过平台判断和响应式约束隔离。

## 非目标

1. 不修改 IM 协议、消息存储、会话同步、音视频信令、推送链路。
2. 不重构所有二级设置页、群管理页、朋友圈、工作台等低频页面。
3. 不引入新的大型 UI 框架。
4. 不为了换肤删除之前已修复的 emoji、撤回、反应、头像、通话记录逻辑。
5. 不在本轮统一全平台视觉，移动端只允许被动继承安全的 token，不改变导航结构。

## 视觉方向

### 风格

温暖社交感，基于干净的白色表面、浅暖页面底色、橙色主行动色、青绿色在线/成功状态、深色正文。整体要比当前灰白界面更有产品感，但避免过度活泼。

UI Pro Max 检索建议里包含 “Vibrant & Block-based” 和高对比社交风格。本项目采用其中的“清晰块面、明确 hover/focus、友好字体层级”，但不采用 App Store landing 或营销页结构，因为 WuKong IM Web 首屏应是实际聊天工作台。

### 颜色 token

建议新增 Web B 主题语义色，保留现有 `WKColors` 兼容别名：

| 角色 | 色值 | 用途 |
| --- | --- | --- |
| pageWarm | `#FFFAF5` | Web 页面背景 |
| surface | `#FFFFFF` | 主要容器、列表、输入区 |
| surfaceSoft | `#FFF7ED` | 搜索框、弱选中、空态背景 |
| borderWarm | `#FED7AA` | Web 边框和分隔线 |
| action | `#F97316` | 发送、主按钮、当前导航 |
| actionSoft | `#FFEDD5` | 当前会话、自己消息气泡 |
| online | `#0D9488` | 在线、成功、可用状态 |
| textPrimary | `#172033` | 标题和正文 |
| textSecondary | `#64748B` | 辅助文字 |
| danger | `#EF4444` | 删除、失败、挂断 |

### 排版

继续使用当前 Flutter 字体体系和 emoji fallback，避免引入外部网络字体。Web 端标题字重更清楚，正文不随视口宽度缩放。

建议层级：

| 层级 | 尺寸 | 用途 |
| --- | --- | --- |
| Web page title | 22 | 会话/通讯录/我的主标题 |
| Panel title | 16 | 列表分组、右侧上下文标题 |
| Body | 14 | 会话摘要、消息文字、表单 |
| Meta | 12 | 时间、状态、徽标 |
| Tiny | 11 | 次级状态、角标辅助 |

### 形状和阴影

主要控件圆角控制在 8px 左右。头像可使用 10px 到 12px 以保持亲和感。页面级区域不做卡片套卡片。阴影只用于弹层、浮动来电、菜单和当前输入区轻微抬升。

## 信息架构

### 宽屏 Web

当 Web 可用宽度达到桌面阈值时，主界面采用三到四区结构：

1. 左侧导航 rail：消息、通讯录、我的，底部放设置/状态入口。宽度约 64px。
2. 会话或目录列表：搜索、筛选、列表、空态。宽度约 300px 到 360px。
3. 主聊天区：标题栏、消息时间线、输入区。占据剩余空间。
4. 右侧上下文区：会话信息、成员/文件/置顶消息，默认可折叠。宽屏显示，窄屏隐藏。

### 窄屏 Web

窄屏 Web 保留单页导航体验：

1. 主壳层保持底部 tabs。
2. 会话列表进入聊天页后隐藏列表。
3. 右侧上下文变成抽屉或独立页面。
4. 输入区不被键盘、安全区、底部 tab 遮挡。

### 断点

| 宽度 | 行为 |
| --- | --- |
| `< 720px` | 移动 Web 单列，底部 tabs |
| `720px - 1023px` | 平板 Web，两列或列表/内容切换 |
| `>= 1024px` | 桌面 Web，左 rail + 列表 + 聊天 |
| `>= 1280px` | 可显示右侧上下文区 |

## 页面规格

### 登录页

登录页保留手机号、密码、记住密码、自动登录、服务端地址入口和协议确认。视觉替换为居中的 Web 登录面板加左侧品牌信息区域。首屏不做营销 hero，重点是登录效率。

状态要求：

1. 加载、错误、禁用、表单校验都有清晰提示。
2. Web 表单宽度固定在舒适范围，移动端自适应。
3. 服务器地址修改入口保留但弱化，不抢主登录按钮。

### 主壳层

`WKTabShell` 需要增加 Web 适配能力。桌面 Web 使用左侧 rail，移动 Web 和非 Web 保持底部 tabs。

导航项应包含图标、可访问标签、当前态、未读角标。当前态使用 `actionSoft` 背景和 `action` 图标/文字。未读角标使用 `danger` 或当前项目已有 badge 语义。

连接状态 banner 保留，但 Web 样式要收敛到顶部窄条，不能遮挡导航或聊天标题栏。

### 会话列表

会话列表采用紧凑但可扫读的结构：

1. 顶部标题和搜索框。
2. 可选分段筛选：全部、未读、群聊、服务号。第一阶段可以先做视觉槽位，不新增筛选业务。
3. 会话行固定高度约 72px 到 80px，避免消息摘要变化导致跳动。
4. 头像、标题、时间、摘要、未读、置顶、免打扰、发送状态、输入中、通话状态都有固定位置。
5. 当前选中会话用 `actionSoft` 和暖色边框强调。

头像要求：

1. 使用既有头像解析结果，不重新猜测用户身份。
2. 缺省头像基于频道 ID 或用户 ID 稳定生成颜色。
3. 群聊、客服、文件助手、系统通知应有稳定识别。

### 聊天页

聊天页需要强化 Web 工作台感：

1. 顶部栏包含返回按钮（窄屏）、头像、标题、在线/成员状态、搜索、语音、视频、更多。
2. 消息时间线使用浅暖背景。消息气泡保持可读，自己消息使用 `actionSoft`，对方消息使用白色。
3. 撤回、系统提示、时间分割、加载更多使用低饱和状态条。
4. 表情反应展示真实 emoji 字符，继续依赖现有 emoji fallback，不允许回退成 `xx`。
5. 长按/右键消息操作统一使用 Web 弹层样式，反应、回复、转发、撤回、复制、删除等入口间距一致。
6. 输入区使用固定高度的工具栏和可增长文本框，附件、表情、语音、发送按钮位置稳定。
7. 拖拽文件到聊天页时，Web drop target 用同一暖色主题。

消息行为约束：

1. 不改变消息发送、重发、撤回、回复、编辑、收藏、转发的业务判断。
2. 不改变音视频通话按钮触发的调用链。
3. 不改变消息列表分页、定位、搜索锚点和缓存策略。

### 通讯录

通讯录使用 Web 列表工作台样式：

1. 左侧目录：搜索、好友、群聊、客服、标签入口。
2. 右侧详情：选中联系人后显示头像、名称、状态、发消息、音视频入口。
3. 窄屏保持当前列表进入详情的模式。

第一阶段可优先统一标题、搜索、列表项、空态和详情卡样式，不新增通讯录业务分类。

### 个人中心

个人中心从大面积旧灰界面调整为设置面板：

1. 顶部用户资料区：头像、昵称、账号、安全状态。
2. 功能分组：账号与安全、通知、通用、帮助与反馈、关于。
3. Web 宽屏下使用两列信息区，窄屏单列。
4. 保留既有入口，不删除功能。

### 来电和通话入口

音视频相关 UI 要完整覆盖：

1. 发起通话后本地应看到拨打中状态和会话内记录入口。
2. 收到来电时 Web 端显示高优先级浮层，不依赖回到会话列表。
3. 浮层包含头像、名称、语音/视频类型、接听、拒绝、静音等状态。
4. 通话失败、取消、拒绝、未接在会话中仍使用统一状态样式展示。

本规格只定义视觉与页面反馈，不改信令实现。

## 核心组件

### WebChatShell

职责：在 Web 桌面宽度下组合 rail、列表、聊天区和右侧上下文。非 Web 或窄屏时回落到现有页面栈。

依赖：`HomeShellPage`、`WKTabShell`、当前 tab 状态、现有页面 widgets。

### WarmConversationTile

职责：替代或扩展 `WKConversationItem` 的 Web 样式，保持现有数据输入。提供稳定高度、选中态、hover、未读、发送状态和头像布局。

### WarmMessageBubble

职责：在 `MessageBubble` 或其上层列表项中统一 Web 气泡、反应条、引用、编辑、失败状态。继续复用现有消息 view model。

### WarmComposer

职责：统一 Web 输入区工具按钮、文本框、表情/附件面板入口、发送按钮、录音状态。不要改变 composer controller。

### WarmPopupSurface

职责：统一消息操作菜单、反应选择器、更多菜单、来电浮层。提供圆角、阴影、焦点、键盘退出和遮罩规范。

## 可访问性和交互

1. 所有可点击元素必须有 tooltip 或语义标签，尤其是 rail 图标、语音、视频、更多、附件、表情、发送。
2. hover、pressed、focused、disabled 状态必须可区分。
3. 文本对比度按 4.5:1 作为目标，辅助文字不得低到不可读。
4. 动效控制在 150ms 到 250ms，并遵守 `MediaQuery.disableAnimations` 或平台 reduced motion 信号。
5. 键盘可达：搜索框、会话行、输入框、发送按钮、弹层操作都能通过 Tab/Enter/Escape 操作。
6. 不使用 emoji 作为功能图标，功能图标继续使用现有 assets 或 Flutter Icons。

## 响应式和布局安全

1. 所有固定格式元素需要稳定尺寸，包括 rail、头像、会话行、输入工具栏、消息操作按钮、未读角标。
2. 长标题、长摘要、长按钮文字必须省略或换行，不允许溢出覆盖相邻内容。
3. Web 宽屏布局不能把聊天内容包进多层卡片。
4. 消息列表滚动性能不能因视觉改造下降，仍需保留现有 cache extent 和分页触发策略。
5. 空态、加载、失败、重试都要使用同一视觉语言。

## 验收标准

功能验收：

1. Web 登录、进入会话列表、进入聊天、发送文字/emoji、撤回、表情回复、音视频入口、来电浮层都可正常使用。
2. 之前修复的问题不回归：emoji 不显示 `xx`，撤回实时刷新，长按反应显示真实 emoji，通话有本地/对端反馈，头像身份稳定。
3. 非 Web 平台导航结构不被强制替换。

视觉验收：

1. 1440px 宽屏显示左 rail、会话列表和聊天区，布局不空、不挤、不遮挡。
2. 375px 移动 Web 保持单列可用，输入区不溢出。
3. 会话行、消息气泡、输入框、弹层、按钮有统一暖色主题。
4. 页面不呈现单一橙色堆叠，青绿、深色文字和白色表面需要形成清晰层次。
5. 无文本重叠、按钮文字溢出、头像错位、未读角标压住标题等问题。

工程验收：

1. 新增或更新 widget 测试覆盖 Web shell、会话项、消息气泡、登录页关键状态。
2. 跑通相关 `flutter test`。
3. 跑通相关 `flutter analyze`。
4. 跑通 `flutter build web --release`。
5. 用浏览器截图检查桌面和移动 Web 关键页面。

## 建议实施顺序

1. Web B tokens 和主题适配。
2. Web 主壳层：`WKTabShell` 桌面 rail、窄屏回退、连接状态 banner。
3. 登录页和基础表单组件统一。
4. 会话列表和会话项 Web 样式。
5. 聊天页顶部栏、消息背景、气泡、反应、输入区、操作弹层。
6. 来电/通话浮层视觉和会话内状态展示。
7. 通讯录和个人中心 Web 样式补齐。
8. 响应式、无障碍、截图和回归测试。

## 风险和处理

1. 风险：当前工作区已有大量未提交改动，UI 改造可能与既有修复交错。
   处理：实施计划阶段按文件分批，先测试后改 UI，避免回滚未知改动。

2. 风险：Web 桌面壳层可能改变现有页面生命周期。
   处理：保留 `IndexedStack` 或等价状态保持机制，避免切 tab 丢状态。

3. 风险：聊天页文件较大，直接改动容易影响业务逻辑。
   处理：把视觉组件拆到小 widget，优先包裹/替换展示层，不移动消息处理逻辑。

4. 风险：中文字符串在部分文件中存在编码显示问题。
   处理：新改动优先使用已有 copy 常量或 Unicode 字符串，避免扩大乱码范围。

5. 风险：过多暖色会导致界面单调。
   处理：主行动用橙色，状态和在线用青绿，正文用深色，表面保持白色和浅暖底。

## 后续流程

用户确认本设计规格后，进入实施计划阶段。实施计划需要拆成可验证的步骤，并在动产品代码前定义测试点。
