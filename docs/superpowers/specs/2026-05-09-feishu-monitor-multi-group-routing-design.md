# 飞书监控多群转发规则设计

## 目标

把当前“所有飞书消息转发到一个悟空 IM 群”的 MVP，升级为“多个飞书来源群分别转发到不同悟空 IM 群”的规则系统。

用户要解决的问题是：

- 需要同时监控多个普通飞书群。
- 不想为了转发某个飞书群而手动打开固定飞书群页。
- 不知道悟空 IM 群 ID 时，也能通过群昵称选择目标群。
- 已解散、不可用、未保存的悟空 IM 群不应出现在目标选择里。
- 长期运行时，未配置规则的飞书群不应误转发到错误群。

## 设计结论

采用“全局捕获 + 本地路由规则”的方案。

```text
Feishu Monitor Shell
  捕获所有可观察到的飞书消息事件
  输出 normalized recentEvents / observedConversations

WuKongIM 飞书信息监控中心
  管理多条转发规则
  按来源飞书群匹配目标悟空 IM 群
  执行手动转发和自动转发
```

MVP 规则：

- 一个飞书来源群最多绑定一个悟空 IM 目标群。
- 一个悟空 IM 群可以接收多个飞书来源群。
- 未匹配到启用规则的飞书消息默认跳过，不转发。
- 手动“转发最近事件”和自动轮询转发使用同一套路由逻辑。

## 为什么选择这个方案

比较过三个方向：

1. 继续保留一个全局目标群。
   - 优点：改动小。
   - 缺点：多个飞书群会混到同一个悟空 IM 群里，无法满足当前需求。

2. 在 Feishu Monitor Shell 里做完整转发规则。
   - 优点：Shell 可以独立转发。
   - 缺点：会让 Shell 重新变成第二套管理后台，和“管理功能进入 WuKongIM”方向冲突。

3. Shell 只负责捕获，WuKongIM 管理和执行本地转发规则。
   - 优点：职责清楚，界面统一，方便使用悟空 IM 群选择器和本地 SDK 发送能力。
   - 缺点：WuKongIM 桌面端需要保持运行，MVP 阶段规则先是本地持久化。

推荐并采用第 3 种。

## 范围

本阶段包含：

- 多条飞书来源群到悟空 IM 群的本地转发规则。
- 从“飞书群组”列表给某个飞书群设置转发目标。
- “转发规则”页查看、启用、停用、编辑、删除规则。
- 手动转发最近事件时按规则分发。
- 自动转发时按规则分发并继续去重。
- 未匹配、重复、失败、成功的结果统计和提示。

本阶段不包含：

- 一个飞书群转发到多个悟空 IM 群。
- 正则、关键词、包含匹配。
- 云端规则同步。
- 多飞书账号。
- 图片、文件、卡片的完整媒体转发策略。
- 批量导入导出规则。

## 数据模型

新增多群转发规则模型：

```dart
class FeishuMonitorForwardingRoute {
  final String id;
  final bool enabled;
  final String sourceConversationId;
  final String sourceConversationName;
  final String sourceConversationType;
  final String targetGroupId;
  final String targetGroupName;
  final DateTime createdAt;
  final DateTime updatedAt;
}
```

字段说明：

- `id`：本地规则 ID。
- `enabled`：是否启用。
- `sourceConversationId`：飞书来源会话 ID，优先用于匹配。
- `sourceConversationName`：飞书来源群名称，用于显示和 ID 缺失时兜底匹配。
- `sourceConversationType`：群聊、单聊、机器人等类型，MVP 主要支持群聊。
- `targetGroupId`：悟空 IM 群 ID，实际发送使用。
- `targetGroupName`：悟空 IM 群昵称，主要用于显示和降低误选风险。
- `createdAt` / `updatedAt`：用于审计和排序。

设置模型从单目标升级为：

```dart
class FeishuMonitorForwardingSettings {
  final bool enabled;
  final List<FeishuMonitorForwardingRoute> routes;
}
```

## 持久化

MVP 使用本地持久化，沿用当前桌面端设置存储风格。

推荐实现：

- 保留 `FeishuMonitorForwardingSettingsStore` 抽象。
- 用 `SharedPreferences` 保存 JSON 数组。
- 存储 key 使用新的版本化 key，例如 `feishu_monitor_forwarding_settings_v2`。
- 读取旧单目标设置时可以迁移为一条兼容规则，但只有在有明确来源飞书群时才自动生成；否则只保留自动转发开关，不生成模糊规则。

这样可以避免旧全局目标群在升级后误接收所有来源群消息。

## 路由匹配

匹配顺序：

1. 如果事件有 `conversationId`，优先匹配启用规则的 `sourceConversationId`。
2. 如果事件没有稳定 ID，再用规范化后的 `conversationName` 匹配 `sourceConversationName`。
3. 如果仍然没有匹配，标记为 `skippedUnmatched`，不转发。

名称规范化规则：

- 去掉首尾空白。
- 合并连续空白。
- 大小写不敏感。

冲突处理：

- MVP 不允许同一个 `sourceConversationId` 存在多条启用规则。
- 从飞书群行点击“设置转发”时，如果已有该来源规则，打开编辑，而不是新建重复规则。
- 如果只有群名匹配但存在多个同名来源，要求用户从“飞书群组”列表里选择具体来源，不自动猜测。

## 转发流程

### 手动转发最近事件

管理员点击“转发最近事件”后：

1. 读取 Shell 当前 `recentEvents`。
2. 读取本地启用规则。
3. 对每条事件执行路由匹配。
4. 未匹配事件跳过。
5. 已匹配事件按目标悟空 IM 群发送。
6. 用事件 `dedupeKey` 防止重复转发。
7. 显示本次结果：成功、重复跳过、未匹配跳过、失败。

### 自动转发

自动转发开启后：

1. 继续按固定间隔刷新 Shell 状态。
2. 对新增 `recentEvents` 应用同一套路由匹配。
3. 未配置规则的飞书群不转发。
4. 单条失败只记录失败，不阻塞其他规则和其他群。
5. Shell 离线或未登录时停止发送，并在状态区显示原因。

- 类型

- 今日事件数

- 目标悟空 IM 群

操作：

- 暂停转发
- 查看最近事件
- 复制 ID

- 弹出目标悟空 IM 群选择器。

- 过滤已解散、未保存、异常状态的群。

### 顶部状态和快捷操作

保留现有：

- Shell 在线状态

- 监听状态
- 今日捕获
- 今日成功
- 今日失败
- 启动监控

## 错误处理

### 未匹配来源群

不发送。

界面显示：

```text
有 3 条事件未匹配转发规则，已跳过。
```

### 目标群无效

规则保留，但标记为异常。

发送时跳过该规则并记录失败原因：

```text
目标悟空 IM 群不可用或已不存在。
```

### Shell 离线

禁用手动转发按钮或点击后显示明确错误。

```text
Feishu Shell 未在线，无法读取最近事件。
```

### Feishu 未登录

保留规则，不转发。

状态区提示需要重新扫码。

### 重名飞书群

如果 ID 存在，以 ID 为准。

如果 ID 缺失并出现重名，不自动匹配，要求用户从飞书群组列表选择具体来源。

## 兼容当前单目标 MVP

当前单目标设置不能直接等价升级为“所有来源都发到这个群”，因为这样会在多群场景下误转发。

兼容策略：

- 保留现有目标群显示为“旧单目标配置”提示。
- 如果当前 Shell 最近事件里只有一个来源群，可以引导用户一键生成一条规则。
- 如果存在多个来源群，要求用户逐个设置规则。
- 新自动转发只认多规则，不再把未匹配消息发到旧目标群。

## 安全和运维

- 规则只保存本地群 ID 和群名，不保存悟空 IM 登录密钥。
- Shell token 仍在系统设置里管理，不进入规则表。
- 日志中可以显示群名和消息摘要，但不输出完整鉴权信息。
- 删除规则只删除本地路由，不删除飞书群或悟空 IM 群。
- 停用规则比删除规则更适合临时暂停。

## 测试策略

### 单元测试

- 多规则 JSON 序列化和反序列化。
- 旧单目标设置迁移行为。
- 按 `conversationId` 匹配。
- 按规范化 `conversationName` 兜底匹配。
- 未匹配事件跳过。
- 重复 `dedupeKey` 不重复发送。
- 禁用规则不发送。
- 目标群为空或无效时失败计数正确。

### Widget 测试

- 飞书群组页显示来源群转发状态。
- 点击“设置转发”自动带入来源群。
- 目标悟空 IM 群选择器显示昵称和 ID。
- 已解散、未保存、异常群不出现在选择器里。
- 转发规则页显示多条规则。
- 启用、停用、删除规则后 UI 更新。

### 手工联合测试

1. 启动 Feishu Monitor Shell。
2. 登录飞书并进入消息页。
3. 在 WuKongIM 飞书信息监控中心启动监控。
4. 等待至少两个飞书群出现在“飞书群组”页。
5. 将飞书群 A 绑定到悟空 IM 群 1。
6. 将飞书群 B 绑定到悟空 IM 群 2。
7. 分别在飞书群 A 和 B 发送新消息。
8. 确认消息分别进入对应悟空 IM 群。
9. 在未绑定飞书群 C 发送消息。
10. 确认群 C 消息不转发，并在结果中显示未匹配跳过。
11. 停用飞书群 A 的规则。
12. 确认飞书群 A 后续消息不再转发。

## 成功标准

- 用户可以不用知道悟空 IM 群 ID，也能通过群昵称设置目标群。
- 用户可以给多个飞书群分别配置不同目标群。
- 手动转发和自动转发都按规则分发。
- 未绑定飞书群不会误转发。
- 转发结果能清楚区分成功、重复、未匹配、失败。
- 当前单目标 MVP 的已验证发送链路继续可用，但不再作为多群场景的默认路由。

## 后续扩展

后续可以在这个规则模型上扩展：

- 云端规则同步。
